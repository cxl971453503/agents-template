# AGENTS.md

## 模板定位
本仓库是一个通用多智能体项目模板，不代表任何具体业务项目。

当用户拿到本模板后，只需要告诉 Codex “我要做一个什么项目”，Agent 应从项目发现开始，通过问答逐步生成项目文档、任务计划、实现与评审结果。

默认禁止把本仓库理解为任务管理系统、Web 应用、Next.js 项目或任何固定技术栈。`examples/` 目录中的内容仅为示例，不是当前项目事实来源。

---

## 核心使命
任何 Agent 在执行任务前，必须先判断当前阶段，并严格遵守阶段边界。

Agent 的核心职责：
- 在正确阶段做正确的事
- 在信息不足时优先澄清，而不是猜测
- 在已有约束下进行最小可行变更
- 让项目文档随着项目演进持续同步

---

## 执行角色
本模板采用 5 个通用执行角色，角色入口位于 `.codex/agents/`。

| 角色 | 配置文件 | 主要职责 |
| --- | --- | --- |
| PM | `.codex/agents/pm.toml` | 项目发现、产品定义、MVP 边界、需求澄清 |
| Explorer | `.codex/agents/explorer.toml` | 阅读上下文、分析现有结构、识别约束与风险 |
| Builder | `.codex/agents/builder.toml` | 任务拆分、文档生成、代码或产物实现 |
| Tester | `.codex/agents/tester.toml` | 测试、审查、质量门禁、风险分级 |
| Reporter | `.codex/agents/reporter.toml` | 阶段交接、变更总结、文档同步检查 |

详细角色说明位于 `docs/agents/`。

## 工作流阶段
- 阶段 0：项目发现
- 阶段 1：产品定义
- 阶段 1.5：体验与界面设计（可选）
- 阶段 2：方案设计
- 阶段 3：任务规划
- 阶段 4：构建实现
- 阶段 5：测试与评审
- 阶段 6：交接与沉淀

默认阶段链路：
- 阶段 0 -> 阶段 1
- 阶段 1 -> 阶段 1.5 / 阶段 2
- 阶段 1.5 -> 阶段 2 / 阶段 4
- 阶段 2 -> 阶段 3
- 阶段 3 -> 阶段 4
- 阶段 4 -> 阶段 5
- 阶段 5 -> 阶段 6

若要跳阶段，必须说明：
- 为什么可以跳
- 缺失了哪些标准输入
- 当前采用了哪些临时假设
- 跳阶段带来的风险

## 首次进入规则
如果以下任一情况成立，必须先进入阶段 0：项目发现。

- `docs/project-overview.md` 不存在、为空或仍是模板占位
- `docs/product-requirements.md` 不存在、为空或仍是模板占位
- `docs/architecture.md` 不存在、为空或仍是模板占位
- 用户只描述了模糊想法，尚未形成稳定项目目标
- 当前项目方向、技术形态或交付物类型不清楚

阶段 0 中，Agent 不应要求用户手动填写完整文档。应通过简短问答引导用户，并在信息足够时自动生成初始文档。

每轮澄清最多提出 3~5 个高价值问题，优先使用选项、示例和对比帮助用户回答。

## 阶段入口检查与退出条件

### 阶段 0：项目发现
入口条件：
- 项目目标、目标用户、交付形态或 MVP 范围尚不明确
- 关键项目文档不存在、为空或仍是模板占位

退出条件：
- 已明确项目目标、目标用户、MVP 范围、非目标范围和约束
- 已生成或更新：
  - `docs/project-overview.md`
  - `docs/product-requirements.md`
  - `docs/architecture.md` 或 `docs/solution-design.md`
- 当前范围与未来扩展已做基本区分
- 剩余不确定点不会阻塞进入阶段 1 或阶段 2

推荐角色：PM

### 阶段 1：产品定义
入口条件：
- 已具备 `docs/project-overview.md`
- 项目目标、用户和 MVP 范围基本明确
- 当前问题主要是“做什么”

退出条件：
- `docs/product-requirements.md` 已明确核心问题、目标用户、MVP 范围、非目标范围和验收标准
- 需求足够支持体验设计或方案设计继续推进
- 没有关键产品问题处于完全未定义状态

推荐角色：PM

### 阶段 1.5：体验与界面设计
入口条件：
- `docs/product-requirements.md` 已存在且不为空
- 项目包含界面、交互、内容结构、用户流程或信息架构
- 当前问题主要是体验结构，而不是技术实现

退出条件：
- `docs/uiux-design.md` 已明确页面/界面列表、用户流程、信息结构、组件或模块拆分建议、状态设计和关键交互规则
- 已参考 `docs/workflow/uiux-review-checklist.md` 检查主体验是否被错误压缩到侧边栏或次级容器
- 设计结果足够支持方案设计或构建实现

推荐角色：PM / Explorer

### 阶段 2：方案设计
入口条件：
- `docs/project-overview.md` 已存在
- `docs/product-requirements.md` 已存在
- 若有复杂交互，优先参考 `docs/uiux-design.md`
- 当前问题主要是模块划分、数据模型、接口、技术方案、内容结构或交付策略

退出条件：
- `docs/architecture.md` 或 `docs/solution-design.md` 已明确系统/方案概览、模块边界、数据或内容模型、关键决策、风险与简化策略
- 如需接口定义，`docs/api-contracts.md` 已具备初稿
- 方案信息已足够支持任务拆解

推荐角色：Explorer

### 阶段 3：任务规划
入口条件：
- 方案文档已存在并可用于拆解任务
- 若涉及接口协作，`docs/api-contracts.md` 已存在或已有明确草案
- 当前问题主要是“怎么拆任务、排依赖、定完成标准”

退出条件：
- `docs/task-board.md` 已明确里程碑、任务列表、任务输入输出、依赖关系和完成定义
- 任务粒度适合单次 Agent 执行
- 构建实现阶段可以拿着具体任务直接开工

推荐角色：Builder

### 阶段 4：构建实现
入口条件：
- 已存在可执行任务，包含 Task ID、目标、输入、输出和完成定义
- 已具备必要方案文档、工程规范、任务看板和相关上下文
- 若涉及接口、复杂交互或测试门槛，对应文档已存在或有明确临时假设

退出条件：
- 当前任务已按既定范围完成
- 变更与方案、契约和工程规范一致
- 已完成必要验证，并明确未覆盖项
- 若影响文档，相关文档已同步更新

推荐角色：Builder

### 阶段 5：测试与评审
入口条件：
- 已存在待审查的代码、文档或其他产物变更
- 已具备测试策略、文档同步规则和评审清单，或已有明确临时评审标准

退出条件：
- 已完成正确性、可维护性、测试覆盖、契约一致性和文档同步检查
- 已记录阻塞问题、重要问题、次要问题和测试缺口
- 已给出明确结论：可通过、需修改后再审、存在风险但可接受

推荐角色：Tester

### 阶段 6：交接与沉淀
入口条件：
- 当前阶段或任务已完成，需要把上下文交给下一阶段、下一角色或用户

退出条件：
- 已说明当前阶段是否达到退出条件
- 已列出下一阶段输入、已更新文档、风险和待确认项
- 已按 `docs/workflow/stage-handoff-template.md` 输出交接说明

推荐角色：Reporter

## 文档生成规则
新项目启动时，优先从 `docs/template/*.template.md` 生成项目专属文档。

常见项目文档：
- `docs/project-overview.md`
- `docs/product-requirements.md`
- `docs/uiux-design.md`（需要界面或体验设计时）
- `docs/architecture.md` 或 `docs/solution-design.md`
- `docs/api-contracts.md`（需要接口协作时）
- `docs/task-board.md`
- `docs/test-strategy.md`

模板文件只作为结构参考，不应被当作真实项目事实。

---

## 上下文读取规则
开始任何任务前，按需读取：
- `AGENTS.md`
- 对应 `.codex/agents/*.toml`
- 对应 `docs/agents/*.md`
- 当前阶段所需项目文档
- `docs/workflow/terminology-and-naming.md`
- `docs/workflow/project-bootstrap-checklist.md`（首次进入或上下文不明时）
- `docs/workflow/adopt-into-existing-project.md`（接入已有项目时）
- `docs/workflow/project-type-guide.md`（判断项目类型时）
- `docs/workflow/engineering-guidelines.md`
- `docs/workflow/document-sync-matrix.md`
- `docs/workflow/skill-selection-guide.md`（涉及外部 Skill 时）

如果文档不存在、不完整或彼此冲突，应先：
1. 指出缺失或冲突的上下文
2. 从已有内容中提炼最小可行假设
3. 若假设风险较高，发起简短澄清问答
4. 在输出中显式标注假设及风险

## 全局规则
1. 一次只完成一个明确任务，不允许无边界扩展。
2. 不要把 `examples/` 中的示例内容当成当前项目需求。
3. 需求阶段只允许输出产品文档，不允许直接写业务代码。
4. 方案阶段只允许输出技术或交付方案，不允许直接跳到完整实现。
5. 构建阶段必须基于任务定义、方案文档和工程规范执行。
6. 测试与评审阶段必须基于测试策略、评审清单和实际变更执行。
7. 发现需求、方案、实现三者冲突时，先列出冲突点，再说明影响，最后给出建议处理方案。
8. 默认优先可维护性、简洁性、复用现有结构和小步修改。
9. 除非任务明确要求，否则不要重构整个项目、替换核心技术栈、引入重基础设施或修改公共契约。
10. 所有 Markdown、TOML 和代码文件默认使用 UTF-8 编码。
11. 引入外部 Skill 前，必须参考 `docs/workflow/skill-selection-guide.md`，说明启用原因、依赖前提、风险和验证方式。
12. 复制模板后首次启动项目时，必须参考 `docs/workflow/project-bootstrap-checklist.md`，确认是否仍处于模板状态。
13. 项目类型不明确时，参考 `docs/workflow/project-type-guide.md` 判断主类型和次要类型。
14. 将本模板接入已有项目时，必须参考 `docs/workflow/adopt-into-existing-project.md`，先保护已有项目上下文，再合并多 Agent 流程。

## 构建阶段要求
构建类任务必须：
- 先说明实现思路
- 列出假设
- 再进行修改
- 最后提供变更摘要、验证结果和未覆盖项
- 对新增或修改的类型、字段、函数补充中文注释
- 对关键业务流程、状态流转和边界判断补充中文说明

如果当前项目不是代码项目，也应将上述要求等价应用到对应产物，例如文档、脚本、配置、数据集或内容资产。

## 输出结构要求
每次输出默认包含：

```md
1. 角色
- 当前角色名称

2. 阶段
- 当前所处阶段

3. 已读取输入
- 已阅读的文档 / 文件 / 上下文

4. 假设
- 当前采用的最小可行假设

5. 输出
- 本轮产出内容

6. 风险
- 当前风险、缺失信息或潜在冲突

7. 下一步
- 建议的下一步动作
```

实现类任务额外包含：

```md
8. 变更摘要
- 本轮实际变更

9. 验证
- 测试结果、验证方式、未覆盖项
```

角色级输出结构可参考 `docs/workflow/role-output-templates.md`。

---

## 阶段交接规则
当一个阶段准备进入下一个阶段时，必须显式说明：
- 当前阶段是否已达到退出条件
- 哪些文档可作为下一阶段输入
- 仍然存在的风险或待确认项
- 建议下一角色

具体交接格式参考 `docs/workflow/stage-handoff-template.md`。
