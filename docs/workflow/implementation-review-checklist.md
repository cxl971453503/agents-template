# 构建评审清单

## 目标
- 为构建类任务提供统一、可重复执行的评审基线。
- 降低多 Agent 协作时遗漏关键检查项的风险。
- 确保评审不只关注“能不能跑”，也覆盖可维护性、契约一致性和可验证性。

## 适用范围
适用于所有阶段 4：构建实现 进入阶段 5：测试与评审的代码、文档、脚本、配置、数据、内容或其他产物。

若评审对象不涉及代码，应跳过代码专属条目，并明确说明跳过原因。

## 使用方式
Tester 在输出审查结论前，应至少逐项检查：

1. 任务范围与完成定义
2. 方案与职责边界
3. 契约与数据 / 内容结构
4. 可读性与注释
5. 错误处理与边界条件
6. 测试与验证
7. 文档同步
8. 风险与后续影响

## 评审清单
### 1. 任务范围与完成定义
- 是否与 `docs/task-board.md` 中当前任务的目标、输入、输出、依赖和完成定义一致
- 是否出现未获批准的需求扩展
- 是否存在无关改动或顺手重构

### 2. 方案与职责边界
- 是否符合 `docs/architecture.md` 或 `docs/solution-design.md` 中的模块划分和边界
- 是否存在入口层承载复杂规则的问题
- 是否存在跨层调用混乱、职责不清的问题
- 是否引入了与当前任务不匹配的复杂基础设施

### 3. 契约与数据 / 内容结构
- 是否符合 `docs/api-contracts.md` 或其他协作契约
- 字段命名、状态值、文件格式、命令参数或错误结构是否一致
- 若契约已变化，是否同步更新调用方、使用方和文档

### 4. 可读性与注释
- 关键变量、字段、函数、流程或文档结构是否能被快速理解
- 所有新建或修改的类型、字段、函数是否都补充了对应的中文注释
- 是否为关键业务规则、状态流转、边界判断补充了中文说明
- 是否存在误导性注释、过期注释或完全缺少说明的复杂逻辑

### 4.5 体验结构合理性
- 如果涉及界面或交互，是否符合 `docs/uiux-design.md`
- 是否违反 `docs/workflow/uiux-review-checklist.md` 中的主体验优先原则
- 是否为了赶 MVP，将核心信息或高频操作压缩到狭窄侧边栏、抽屉或其他次级容器中

### 5. 错误处理与边界条件
- 是否处理了空值、非法输入、资源不存在、权限不足、文件缺失、外部服务失败等典型边界
- 是否存在静默失败、模糊错误或错误分支遗漏
- 是否暴露了不应暴露的内部实现细节

### 6. 测试与验证
- 是否有与本次改动相匹配的测试或验证说明
- 是否覆盖关键成功路径
- 是否覆盖关键失败路径或边界路径
- 未覆盖项是否已明确说明

### 7. 文档同步
- 若改动涉及项目范围、方案、契约、任务边界或测试策略，是否同步更新相关文档
- 若无需更新，是否有合理说明
- 同步判断参考 `docs/workflow/document-sync-matrix.md`

### 8. 风险与后续影响
- 是否存在临时实现、兼容逻辑、技术债或潜在回归风险
- 是否识别出当前可接受但后续必须处理的问题
- 是否已给出明确建议：通过、修改后再审，或带风险通过

## 建议输出格式
```md
1. 总结
- 本次评审对象
- 总体结论

2. 清单检查结果
- 通过项：
- 未通过项：
- 跳过项：

3. 阻塞问题
- 必须修改后才能通过的问题

4. 重要问题
- 不一定阻塞，但风险较高的问题

5. 次要问题
- 可后续优化的问题

6. 测试缺口
- 缺失的测试或验证

7. 处理建议
- 可通过 / 修改后再审 / 带风险通过
```

## 通过标准
默认应满足：
- 无阻塞问题
- 契约与职责边界未被破坏
- 新增或修改的重要定义具备必要中文注释
- 已完成必要测试或明确说明验证范围
- 文档同步检查已完成
