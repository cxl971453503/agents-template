# 工程与产物规范

## 目标
- 为构建阶段提供统一的执行规范。
- 降低多 Agent 协作时的理解偏差和实现漂移。
- 在保证交付速度的前提下维持可维护性、可验证性和一致性。

## 适用范围
本规范适用于代码、文档、脚本、配置、数据处理、内容资产、设计说明等项目产物。

构建类任务默认按需参考：
- `docs/project-overview.md`
- `docs/product-requirements.md`
- `docs/uiux-design.md`（如果存在）
- `docs/architecture.md` 或 `docs/solution-design.md`
- `docs/api-contracts.md`（如果存在）
- `docs/task-board.md`
- `docs/test-strategy.md`
- `docs/workflow/document-sync-matrix.md`
- `docs/workflow/terminology-and-naming.md`

## 核心原则
1. 小步修改，优先完成当前任务，不做无边界扩展。
2. 优先复用现有结构和模式，不重复发明新实现方式。
3. 先保证正确性和一致性，再考虑局部优化。
4. 明确产物边界，避免一个文件、模块或文档承载过多职责。
5. 所有实现都应能被测试、审查或人工验收。

## 任务执行要求
### 开始前
- 确认当前任务对应的 Task ID、目标、输入、输出和完成定义。
- 若输入文档缺失、为空或彼此冲突，先记录问题，再基于最小可行假设推进。
- 高风险假设必须在输出中显式标注。

### 执行中
- 一次只解决一个明确任务。
- 不顺手修复无关问题，不顺手重构相邻模块。
- 不在未更新文档前擅自修改已确认接口、数据结构、交付范围或协作契约。
- 若发现结构性问题，先记录影响范围和建议，不直接做大范围修改。

### 完成后
- 说明改动内容、验证方式、未覆盖项和已知风险。
- 若变更影响项目范围、方案、接口、测试或任务边界，必须同步更新相关文档。

## 通用分层建议
不同项目形态可以有不同分层，但应保持职责清晰。

### 入口层
- 面向用户、命令、页面、接口或外部调用。
- 负责接收输入、展示反馈、组织流程。
- 不承载复杂业务规则。

### 规则层
- 负责业务规则、状态流转、校验、权限、约束和核心判断。
- 应尽量可测试、可复用、可审查。

### 数据 / 内容层
- 负责数据读取、写入、转换、存储、文件处理或内容管理。
- 不混入界面语义或展示逻辑。

### 集成层
- 负责第三方服务、外部 API、模型、工具链或系统边界。
- 应收口错误和兼容性处理，避免外部细节扩散。

## 命名与风格
- 优先使用清晰、直白、可搜索的命名。
- 同一概念在文档、代码、接口和任务中保持一致命名。
- 避免隐式副作用和跨层直接调用。
- 新增公共能力前，先确认现有结构中是否已有相同模式。

## 注释与可读性
### 基本原则
- 优先用准确命名、清晰类型和合理拆分表达意图。
- 注释应解释“为什么这样做”或“这段逻辑在业务上意味着什么”。
- 默认使用中文注释，除非外部协议、标准库或第三方约束必须保留英文表达。

### 默认要求
- 新增或修改的类型定义，应补充中文注释。
- 新增或修改的字段定义，应补充中文注释。
- 新增或修改的函数、方法、命令入口或工作流步骤，应补充中文注释。
- 核心业务流程、状态流转、权限判断、兼容逻辑和边界处理，应补充中文说明。

### 不推荐的注释
- 仅重复代码表面含义的注释。
- 与当前实现不一致或容易过期的注释。
- 大段无结构说明，影响阅读节奏。

## 契约与同步
- 能通过类型、结构或格式表达的约束，不只靠口头说明。
- 接口、文件格式、命令参数、事件、状态值和字段语义发生变化时，必须同步相关文档。
- 具体同步判断参考 `docs/workflow/document-sync-matrix.md`。

## 测试与验证
根据项目形态选择合适验证方式：
- 静态检查：格式、类型、lint、配置校验。
- 单元测试：纯函数、核心规则、边界判断。
- 集成测试：模块协作、接口契约、数据流。
- 端到端测试：真实用户路径、命令链路或交付流程。
- 人工验收：视觉、内容、体验、业务规则或结果质量。

构建类任务完成后，至少说明：
- 已执行的测试或验证
- 未执行的测试或验证
- 未覆盖的风险点

## 禁止事项
- 未经明确要求重构整个项目。
- 未经确认替换核心技术栈或交付形态。
- 未经文档同步修改公共契约。
- 为局部问题引入过重基础设施。
- 把示例项目内容当作当前项目事实。
