# 术语表与命名约定

## 目标
- 统一本模板中的高频术语写法与含义。
- 降低多 Agent 协作时同一概念被多种名称表达的混乱。
- 提高文档、任务、评审与实现输出的一致性。

## 使用原则
- 优先使用本文档中的推荐写法。
- 涉及通用技术缩写时，可以保留英文缩写，但上下文应清楚。
- 同一项目中，不应对同一概念混用多种叫法。
- 示例项目中的术语不得自动成为当前项目术语。

## 核心术语
| 术语 | 推荐写法 | 说明 |
| --- | --- | --- |
| MVP | `MVP` / `MVP 范围` | 最小可行版本，不等于最终完整产品 |
| Non-goals | `非目标范围` | 当前版本明确不做的内容 |
| Acceptance Criteria | `验收标准` | 判断需求或任务是否达成的条件 |
| Task Board | `任务看板` | 任务列表、里程碑、依赖与完成定义所在文档 |
| Project Discovery | `项目发现` | 从模糊想法澄清项目目标与范围的阶段 |
| Product Definition | `产品定义` | 形成需求文档的阶段 |
| Experience Design | `体验与界面设计` | 页面、流程、信息结构、状态与交互设计 |
| Solution Design | `方案设计` | 模块、分层、数据/内容模型、接口或交付方案设计 |
| Task Planning | `任务规划` | 将方案拆成里程碑与可执行任务 |
| Build | `构建实现` | 基于任务进行代码、文档、配置、脚本或内容产物变更 |
| Review & QA | `测试与评审` | 对产物进行审查、测试和质量判断 |
| Handoff | `交接与沉淀` | 阶段结束后向下一阶段提供上下文 |
| API Contract | `接口与协作契约` | API、命令、文件格式、事件或跨角色协作约定 |
| Definition of Done | `完成定义` | 任务完成必须满足的标准 |

## 阶段命名
统一写法：
- `阶段 0：项目发现`
- `阶段 1：产品定义`
- `阶段 1.5：体验与界面设计`
- `阶段 2：方案设计`
- `阶段 3：任务规划`
- `阶段 4：构建实现`
- `阶段 5：测试与评审`
- `阶段 6：交接与沉淀`

## 角色命名
统一写法：
- `PM`
- `Explorer`
- `Builder`
- `Tester`
- `Reporter`

不建议在默认模板中使用强技术工种作为入口角色，例如前端工程师、后端工程师、架构师。项目需要时，可在具体任务中临时指定专业视角。

## 文档命名
- 项目专属文档放在 `docs/` 根层。
- 模板文档放在 `docs/template/`。
- 工作流规则放在 `docs/workflow/`。
- 示例项目放在 `examples/`。
- 文件名优先使用英文 kebab-case，内容标题可使用中文。

## 任务命名
任务标题应以动作开头，描述明确结果，例如：
- `生成项目需求文档`
- `设计导入文件契约`
- `实现数据清洗脚本`
- `补充核心流程测试`

避免模糊标题，例如：
- `处理一下`
- `优化项目`
- `改页面`

## 输出字段命名
统一使用：
- `角色`
- `阶段`
- `已读取输入`
- `假设`
- `输出`
- `风险`
- `下一步`
- `变更摘要`
- `验证`

## 风险与问题命名
- `阻塞问题`：必须处理，否则不能继续或不能通过评审
- `重要问题`：不一定阻塞，但风险较高
- `次要问题`：可后续优化的问题
- `测试缺口`：尚未覆盖或验证不足的部分
