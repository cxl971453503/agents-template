# 工程规范

## 目标
- 为实现阶段提供统一的工程执行规范
- 降低多 Agent 协作时的理解偏差和实现漂移
- 在保证交付速度的前提下维持可维护性、可测试性和一致性

## 适用范围
- 适用于本项目的前端、后端、数据库、测试和文档更新工作
- 实现类任务默认必须同时参考：
  - `docs/architecture.md`
  - `docs/api-contracts.md`
  - `docs/task-board.md`
  - `docs/test-strategy.md`
  - `docs/uiux-design.md`（如果存在）
  - `docs/terminology-and-naming.md`

## 核心原则
1. 小步修改，优先完成当前任务，不做无边界扩展。
2. 优先复用现有结构和模式，不重复发明新实现方式。
3. 先保证正确性和一致性，再考虑局部优化。
4. 页面层、服务层、数据访问层职责分离，避免耦合蔓延。
5. 所有实现都应能被测试和验证，而不是依赖口头假设。

## 任务执行要求
### 开始前
- 先确认当前任务对应的 Task ID、输入、输出和完成定义。
- 若输入文档缺失、为空或彼此冲突，先记录问题，再基于最小可行假设推进。
- 高风险假设必须在输出中显式标注。

### 实现中
- 一次只解决一个明确任务。
- 不顺手修复无关问题，不顺手重构相邻模块。
- 不在未更新文档前擅自修改已确认接口或数据契约。
- 若发现结构性问题，先记录影响范围和建议，不直接做大范围改造。

### 完成后
- 说明改动内容、验证方式、未覆盖项和已知风险。
- 若代码变更影响架构、接口、测试或任务边界，必须同步更新相关文档。

## 推荐目录与职责分层
本项目采用单体全栈模式，推荐保持以下职责边界：

### 页面层
- 负责页面布局、交互组织、数据展示和用户反馈。
- 不直接编写复杂业务规则。
- 不直接在页面中写 Prisma 查询。

### API / Route Handler / Server Action 层
- 负责请求解析、鉴权、调用服务层、格式化响应。
- 不承载复杂业务流程拼装。

### 服务层
- 负责业务规则、状态流转、权限判断、跨实体协作。
- 所有写操作优先经由服务层统一处理。

### 数据访问层
- 负责 Prisma 查询、持久化和与数据库交互。
- 不在数据访问层混入页面逻辑或 UI 语义。

## 前端实现规范
### 页面与组件
- 页面负责组装，组件负责复用。
- 通用组件与业务组件应分离，避免一个组件承担过多上下文。
- 所有关键页面必须明确处理：
  - loading
  - empty
  - error

### 表单与交互
- 表单需同时处理前端校验和服务端错误反馈。
- 成功、失败、提交中状态必须可感知。
- 对关键操作提供明确反馈，避免静默失败。

### 数据展示
- 列表、详情、筛选、状态展示应与 `docs/uiux-design.md` 保持一致。
- 未在设计文档中定义的新交互，不应在实现阶段随意扩展。

## 后端实现规范
### API 设计
- 所有接口默认返回 JSON。
- 响应结构应遵循 `docs/api-contracts.md`：
  - 成功：`{ success: true, data }`
  - 失败：`{ success: false, error }`
- 不应在同一项目中混用多套响应格式。

### 输入校验
- 所有外部输入都必须校验，不能默认信任客户端。
- 对缺失字段、格式错误、非法枚举值返回明确错误码和消息。
- `PATCH` 接口支持部分更新，但仍需校验传入字段是否合法。

### 错误处理
- 优先返回可识别的业务错误码，而不是模糊报错。
- 禁止将数据库错误、堆栈或内部实现细节直接暴露给客户端。
- 未识别异常统一收口为 `INTERNAL_SERVER_ERROR`。

### 业务边界
- 状态流转、权限判断、关联校验等逻辑应放在服务层。
- 不要把业务规则散落在页面、路由和数据库查询调用中。

## 数据与 Prisma 规范
### 模型设计
- 字段命名保持清晰、稳定、一致，避免含义模糊的缩写。
- 枚举字段优先使用有限值集合，避免自由字符串。
- 新增字段时优先考虑默认值、可空性和迁移成本。

### 查询与写入
- 避免在多个位置复制相同查询逻辑。
- 写操作前校验关联资源是否存在，例如：
  - `projectId`
  - `taskId`
  - `assigneeId`
- 对高频筛选字段预留后续索引优化空间，但 MVP 阶段不过早复杂化。

## 命名与代码风格
- 优先使用清晰、直白、可搜索的命名。
- 一个函数只做一类事，一个模块只承载一组稳定职责。
- 避免超长函数、隐式副作用和跨层直接调用。
- 优先复用现有工具函数、类型和错误码定义。
- 新增公共能力前，先确认现有代码中是否已有相同模式。

## 注释与可读性规范
### 目标
- 让后续接手的开发者或 Agent 能快速理解变量、字段、函数和业务流程的作用。
- 降低“代码能跑，但难以维护”的风险。
- 让不了解当前上下文的人也能一眼看出定义内容的职责和语义。

### 基本原则
- 优先用准确命名、清晰类型和合理拆分表达意图，不用注释掩盖糟糕设计。
- 注释应解释“为什么这样做”或“这段逻辑在业务上意味着什么”，而不只是重复代码字面意思。
- 对后续维护成本高、业务语义强或容易误解的代码，必须补充注释。
- 默认使用中文注释，除非外部协议、标准库或第三方约束必须保留英文表达。

### 默认强制要求
- 所有新建或修改的类型定义，都应补充中文注释。
- 所有新建或修改的字段定义，都应补充中文注释。
- 所有新建或修改的函数、方法，都应补充中文注释。
- 若存在自动生成代码、第三方映射文件或框架约束位置不适合加注释，应在变更说明中明确说明原因。

### 必须添加注释的场景
- 新增或修改的类型、接口、枚举、数据模型、配置结构。
- 新增或修改的字段、属性、状态值、布尔开关、配置项。
- 新增或修改的函数、方法、hook、服务层入口、工具函数。
- 核心业务流程的入口函数或服务层主流程。
- 含有业务规则、状态流转、权限判断、兼容逻辑的代码块。
- 字段含义不直观、存在约束条件或取值语义的结构定义。
- 临时规避方案、历史兼容逻辑、外部系统限制。
- 看起来“可以简化”但实际上不能轻易改动的实现。

### 推荐添加注释的场景
- 较长函数的关键步骤分段。
- 复杂条件分支的判定意图。
- 前端页面中的状态切换、数据映射和交互副作用。
- 后端接口中的校验顺序、错误分支和事务边界。
- 变量名虽然可读，但业务来源不明显的关键中间值。

### 不推荐的注释
- 仅重复代码表面含义的注释，例如“给变量赋值”“循环数组”。
- 与当前实现不一致、容易过期的注释。
- 大段无结构的说明文字，影响阅读节奏。

### 具体要求
- 新增或修改函数、方法时，应至少说明：
   - 函数职责
   - 关键输入输出
   - 重要副作用或限制条件
 - 新增或修改字段、属性时，应至少说明：
   - 字段业务含义
   - 是否必填
   - 取值范围或特殊值语义
 - 新增或修改类型、接口、枚举、模型时，应至少说明：
   - 该定义代表什么实体或概念
   - 与其他结构的关系或使用场景
 - 注释应尽量紧贴定义位置，避免读者来回跳转查找。
 - 当变量、函数或字段仅靠命名仍不足以让他人快速理解时，必须补充简短中文注释。
 - 评审时应将“是否所有定义项都具备必要中文注释、是否存在误导性注释”作为检查项之一。

## 类型与契约
- 前后端共享的数据结构应尽量保持一致命名。
- 能通过类型表达的约束，不只靠注释说明。
- 接口字段、数据库字段、前端消费字段之间要避免语义漂移。
- 契约变更必须同步更新：
  - `docs/api-contracts.md`
  - 相关测试
  - 调用方实现

## 测试要求
### 单元测试
- 覆盖纯业务逻辑、输入校验、状态流转和边界条件。

### 集成测试
- 覆盖关键 API 的输入输出、数据库读写和主要失败路径。

### E2E 冒烟测试
- 至少覆盖一条核心用户链路：
  - 登录
  - 创建项目
  - 创建任务
  - 更新状态
  - 添加评论

### 最低验证要求
- 实现类任务完成后，至少说明：
  - 已执行的测试
  - 未执行的测试
  - 未覆盖的风险点

## 文档同步规则
出现以下情况时，必须同步更新文档：

- 具体同步判断应参考 `docs/document-sync-matrix.md`

### 更新 `docs/architecture.md`
- 新增核心模块
- 调整分层边界
- 改变关键技术决策

### 更新 `docs/api-contracts.md`
- 新增接口
- 修改请求或响应字段
- 新增或调整错误码

### 更新 `docs/task-board.md`
- 新任务被拆分、合并或调整依赖关系
- 原任务范围发生变化

### 更新 `docs/test-strategy.md`
- 新增关键链路
- 测试层次或发布门槛发生变化

## 禁止事项
- 未经明确要求重构整个项目
- 未经确认替换核心技术栈
- 未经文档同步修改公共契约
- 在页面层直接写复杂 Prisma 逻辑
- 为局部问题引入过重的基础设施

## 当前范围与未来扩展
### 当前范围
- 单体全栈架构
- 面向 MVP 的简化认证与协作能力
- 优先保证任务主链路可交付

### 未来扩展
- 更细粒度权限模型
- 通知、审计、分页与查询优化
- 更复杂的工作区协作能力

## 完成定义
一个实现类任务完成，至少应满足：
- 目标功能按任务定义完成
- 实现符合架构、接口契约和本规范
- 关键成功路径可验证
- 已知风险和未覆盖项已明确说明
- 如有必要，相关文档已同步更新
