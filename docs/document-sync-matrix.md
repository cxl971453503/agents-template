# 文档同步矩阵

## 目标
- 明确代码和设计变更发生后，必须同步更新哪些项目文档。
- 降低“代码已变化，但文档仍停留在旧状态”的风险。
- 为实现阶段与评审阶段提供统一的文档回写依据。

## 使用方式
当某次任务完成后，先判断本次改动属于哪一类，再按矩阵检查是否需要更新对应文档。

若答案为“需要”，则应：
- 更新对应文档
- 或在实现说明 / 评审说明中明确写出“本次无需更新”的原因

## 同步矩阵
| 变更类型 | 需要同步的文档 | 判断标准 | 常见示例 |
| --- | --- | --- | --- |
| 项目目标、范围、约束变化 | `docs/project-overview.md` | 项目方向、目标用户、MVP 范围、非目标范围、约束发生变化 | 从单用户工具调整为团队协作工具 |
| 产品需求变化 | `docs/product-requirements.md` | 功能范围、用户故事、验收标准、非目标范围变化 | 新增任务优先级功能 |
| 页面结构或交互变化 | `docs/uiux-design.md` | 页面结构、用户流程、组件拆分、状态设计变化 | 列表页改为抽屉式详情交互 |
| UI 结构评审基线变化 | `docs/uiux-review-checklist.md` | 对主视区、侧边栏、信息可见性、MVP 结构妥协边界的判断标准变化 | 新增“游戏主系统不得塞入侧边栏”规则 |
| 模块边界或技术方案变化 | `docs/architecture.md` | 模块职责、分层方式、数据流、关键技术决策变化 | 引入服务层、调整模块边界 |
| 接口或错误码变化 | `docs/api-contracts.md` | 请求字段、响应结构、状态值、错误码变化 | 新增接口、修改响应格式 |
| 任务拆分或范围变化 | `docs/task-board.md` | 任务输入、输出、依赖、完成定义变化 | T-104 被拆成前后端两个子任务 |
| 测试层次或发布门槛变化 | `docs/test-strategy.md` | 新增关键链路、改变测试优先级、调整发布门槛 | 将评论流加入必测链路 |
| 实现与评审规则变化 | `docs/engineering-guidelines.md`、`docs/implementation-review-checklist.md` | 工程规范、注释要求、评审基线变化 | 新增代码注释规范 |

## 实现阶段最常见的同步判断
### 只改页面样式或组件布局
- 通常不需要更新 `docs/architecture.md`
- 若交互流程、状态设计、信息结构变化明显，则需要更新 `docs/uiux-design.md`

### 改接口实现但不改契约
- 通常不需要更新 `docs/api-contracts.md`
- 仍应确认 `docs/architecture.md` 是否因分层方式变化需要更新

### 改接口契约或字段语义
- 必须更新 `docs/api-contracts.md`
- 若影响任务范围或测试重点，也应同步检查：
  - `docs/task-board.md`
  - `docs/test-strategy.md`

### 新增业务规则或状态流转
- 先检查是否影响：
  - `docs/product-requirements.md`
  - `docs/architecture.md`
  - `docs/test-strategy.md`
- 若规则属于用户可感知行为，通常也应检查 `docs/uiux-design.md`

### 只补测试
- 一般不需要更新产品、架构、接口文档
- 若测试范围或发布门槛发生变化，需要更新 `docs/test-strategy.md`

## 实现者自查问题
- 这次改动有没有改变“系统怎么工作”
- 这次改动有没有改变“接口怎么调用”
- 这次改动有没有改变“任务原本要交付什么”
- 这次改动有没有改变“什么才算通过测试”
- 这次改动有没有改变“用户实际会看到什么”

若其中任一答案为“有”，则至少应检查一次相关文档是否需要同步。

## 评审者检查要求
- Reviewer 不应只检查代码，还应检查本次改动是否遗漏文档同步。
- 若代码改动明显影响契约、架构、流程或测试，但文档未更新，默认应记为问题。
- 若作者声明“无需更新文档”，应判断该说明是否合理。

## 备注
- 文档同步矩阵用于降低遗漏，不要求每次都更新所有文档。
- 关键目标是：文档与当前实现保持可用的一致性，而不是制造额外文档负担。
