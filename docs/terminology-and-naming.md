# 术语表与命名约定

## 目标
- 统一本模板中的高频术语写法与含义。
- 降低多 Agent 协作时同一概念被多种名称表达的混乱。
- 提高文档、任务、评审与实现输出的一致性与可读性。

## 使用原则
- 优先使用本文件中约定的中文术语。
- 涉及通用技术缩写时，可保留英文缩写，但首次出现时应保证上下文清晰。
- 同一份文档中，不应对同一概念混用多种叫法。

## 核心术语
| 术语 | 推荐写法 | 说明 |
| --- | --- | --- |
| MVP | `MVP` / `MVP 范围` | 指最小可用版本，不建议写成“最小产品”这种容易歧义的说法 |
| API | `API` / `API 契约` | 指接口及其请求、响应、错误码约定 |
| E2E | `E2E` / `E2E 冒烟测试` | 指端到端测试，优先与“单元测试”“集成测试”并列使用 |
| UI/UX | `UI/UX 设计` | 指页面结构、交互、信息架构与状态设计 |
| DoD | `完成定义` | 不再单独使用 `DoD`，统一写为“完成定义” |
| Acceptance Criteria | `验收标准` | 指需求或任务达成的判定条件 |
| Non-goals | `非目标范围` | 指当前明确不做的内容 |
| User Story | `用户故事` | 指以用户视角描述需求价值的表达方式 |
| Task Board | `任务看板` | 指任务列表、里程碑、依赖与完成定义所在文档 |
| Project Discovery | `项目发现` | 指从模糊想法澄清项目目标与范围的阶段 |
| Product Definition | `产品定义` | 指形成需求文档的阶段 |
| Architecture Design | `架构设计` | 指形成模块、分层、数据模型、接口草案的阶段 |
| Task Planning | `任务规划` | 指将架构拆成里程碑与任务的阶段 |
| Implementation | `实现` | 指基于任务与文档进行实际代码变更的阶段 |
| Review & QA | `评审与质量保证` | 指对代码或文档进行审查与验证的阶段 |
| Service Layer | `服务层` | 指承载业务规则、状态流转、权限判断的层 |
| Route Handler / Server Action | `路由处理层` / `Server Action` | 若需中文解释可写“路由处理层”，保留具体框架术语时可直接写英文 |
| Data Model | `数据模型` | 指实体结构、字段设计与关系定义 |
| Error Code | `错误码` | 指接口或业务逻辑返回的机器可识别错误标识 |
| Handoff | `阶段交接` | 指阶段结束后向下一阶段提供的标准化说明 |

## 命名约定
### 文档命名
- 文档标题优先使用中文。
- 文件名可保留英文 kebab-case 或现有命名方式，不强制改动。
- 文档标题应优先体现“文档用途”，例如：
  - `项目概览`
  - `产品需求文档`
  - `架构设计`
  - `测试策略`

### 阶段命名
- 统一写法：
  - `阶段 0：项目发现`
  - `阶段 1：产品定义`
  - `阶段 1.5：UI/UX 设计`
  - `阶段 2：架构设计`
  - `阶段 3：任务规划`
  - `阶段 4：实现`
  - `阶段 5：评审与质量保证`
- 不建议在正式输出中混写：
  - `Stage 4`
  - `Implementation`
  - `开发阶段`
  除非是在引用现有外部术语或代码上下文。

### 任务命名
- 任务标题应以动作开头，描述明确结果，例如：
  - `实现任务创建表单`
  - `新增任务评论接口`
  - `补充任务状态流转测试`
- 避免使用模糊标题，例如：
  - `处理任务`
  - `优化一下页面`
  - `改后端`

### 输出字段命名
- 统一使用：
  - `角色`
  - `阶段`
  - `已读取输入`
  - `假设`
  - `输出`
  - `风险`
  - `下一步`
  - `变更摘要`
  - `验证`
- 不建议在同一项目中混用中英文版字段名。

### 风险与问题命名
- `阻塞问题`：必须处理，否则不能继续或不能通过评审
- `重要问题`：不一定阻塞，但风险较高
- `次要问题`：可后续优化的问题
- `测试缺口`：尚未覆盖或验证不足的部分

## 使用建议
- 新增文档或规则前，先检查是否已有对应术语。
- 若需要引入新术语，优先采用：
  - 易理解
  - 可复用
  - 与现有文档风格一致
  的表达方式。
- 若某术语必须保留英文，应避免再创造第二个中文同义词。
