# 实现评审清单

## 目标
- 为实现类任务提供统一、可重复执行的评审基线。
- 降低多 Agent 协作时遗漏关键检查项的风险。
- 确保评审不只关注“能不能跑”，也覆盖可维护性、契约一致性和可读性。

## 适用范围
- 适用于所有 Stage 4 实现任务进入 Stage 5 评审与质量保证阶段时的代码审查。
- 若评审对象主要是文档，可跳过纯代码相关条目，但必须明确说明跳过原因。

## 使用方式
评审者在输出审查结论前，应至少逐项检查以下内容：

1. 任务范围与完成定义
2. 架构与分层边界
3. 接口与数据契约
4. 代码可读性与注释
5. 错误处理与边界条件
6. 测试与验证
7. 文档同步
8. 风险与后续影响

评审输出中应明确标注：
- 已检查项
- 未通过项
- 跳过项及原因

## 评审清单
### 1. 任务范围与完成定义
- 是否与 `docs/task-board.md` 中当前任务的目标、输入、输出、依赖和完成定义一致
- 是否出现未获批准的需求扩展
- 是否存在“顺手重构”或无关改动

### 2. 架构与分层边界
- 是否符合 `docs/architecture.md` 中的模块划分和边界
- 是否存在页面层直接承载复杂业务逻辑
- 是否存在 route handler 直接堆叠业务逻辑
- 是否存在跨层调用混乱、职责不清的问题

### 3. 接口与数据契约
- 是否符合 `docs/api-contracts.md` 的请求、响应和错误码约定
- 字段命名、状态值、错误结构是否一致
- 若契约已变化，是否同步更新调用方与文档

### 4. 代码可读性与注释
- 关键变量、字段、函数和流程是否能被快速理解
- 所有新建或修改的类型、字段、函数是否都补充了对应的中文注释
- 是否为关键业务规则、状态流转、边界判断补充了中文说明
- 是否存在误导性注释、过期注释或完全缺少注释的复杂逻辑

### 5. 错误处理与边界条件
- 是否处理了空值、非法输入、资源不存在、权限不足等典型边界
- 是否存在静默失败、模糊错误或错误分支遗漏
- 是否暴露了不应暴露的内部实现细节

### 6. 测试与验证
- 是否有与本次改动相匹配的测试或验证说明
- 是否覆盖关键成功路径
- 是否覆盖关键失败路径或边界路径
- 未覆盖项是否已明确说明

### 7. 文档同步
- 若改动涉及架构、接口、任务边界、测试策略，是否同步更新：
  - `docs/architecture.md`
  - `docs/api-contracts.md`
  - `docs/task-board.md`
  - `docs/test-strategy.md`
- 若无需更新，是否有合理说明

### 8. 风险与后续影响
- 是否存在临时实现、兼容逻辑、技术债或潜在回归风险
- 是否识别出当前可接受但后续必须处理的问题
- 是否已给出明确建议：通过、修改后再审，或带风险通过

## 建议输出格式
```md
1. 总结
- 本次评审对象
- 总体结论

2. 清单检查结果
- 通过项：
- 未通过项：
- 跳过项：

3. 阻塞问题
- 必须修改后才能通过的问题

4. 重要问题
- 不一定阻塞，但风险较高的问题

5. 次要问题
- 可后续优化的问题

6. 测试缺口
- 缺失的测试或验证

7. 处理建议
- 可通过 / 修改后再审 / 带风险通过
```

## 通过标准
默认应满足以下条件才可判定为“可通过”：
- 无阻塞问题
- 契约与架构边界未被破坏
- 所有新建或修改的类型、字段、函数具有对应的中文注释
- 已完成必要测试或明确说明验证范围

## 备注
- 本清单用于降低遗漏，不替代专业判断。
- 若项目未来引入自动化检查，本清单应与自动化规则保持一致。
