# UI/UX 评审清单

## 目标
- 在 UI/UX 阶段和实现评审阶段，尽早发现布局结构性问题。
- 防止为了快速跑通 MVP，将核心功能、核心信息或主体验错误地压缩到侧边栏、抽屉或其他次级容器中。
- 特别适用于游戏、模拟器、数据密集型工具、运营后台、复杂工作台等界面。

## 使用时机
- 阶段 1.5：UI/UX 设计完成前
- 阶段 4：实现前，确认布局方案仍然成立
- 阶段 5：评审与质量保证阶段，检查实现是否偏离设计目标

## 核心判断问题
1. 用户一进入页面，是否能直接看到主系统的大部分关键状态
2. 主体验是否占据了页面的主视区，而不是被导航或边栏挤压
3. 被放入侧边栏、抽屉、弹窗的内容，是否真的是次级内容
4. 当前 MVP 简化是否伤害了产品核心体验

## 评审清单
### 1. 主体验优先级
- 页面主视区是否优先承载主玩法、主任务流或主信息流
- 用户进入页面后，是否无需额外展开侧边栏即可理解当前局面
- 主要操作区域是否足够大，能支撑连续使用，而不是只适合点击一两次

### 2. 核心信息可见性
- 核心状态、关键数值、重要反馈是否默认可见
- 用户是否需要频繁打开侧边栏才能了解系统主状态
- 页面首屏是否能表达“现在发生了什么、接下来该做什么”

### 3. 侧边栏与次级容器边界
- 侧边栏是否只承载导航、筛选、辅助信息、低频设置等次级内容
- 抽屉、弹窗、折叠菜单是否只承载非主流程内容
- 是否存在“核心信息放边栏、主视区反而空”的结构性问题

### 4. 游戏 / 数据密集型产品专项检查
- 若产品是游戏、模拟器、工作台、运营盘或策略系统，主系统信息是否集中在主视区
- 用户是否能持续观察关键状态变化，而不是通过切换面板断续查看
- 影响判断和决策的主要信息，是否默认可见而不是隐藏

### 5. MVP 简化合理性
- 当前 MVP 简化是否只是减少功能，而不是破坏核心体验结构
- 是否出现“为了先做出来，把主系统塞进侧边栏”的做法
- 是否记录了暂时接受的 UI 债务，以及后续必须回收的调整点

### 6. 演进与回看
- 在进入实现前，是否重新验证过布局仍然适配当前产品方向
- 若实现中功能密度增加，是否触发了 UI 结构重评估
- 若用户或开发者明确指出布局妨碍理解，是否要求回到 UI/UX 阶段修正

## 通过标准
- 主体验清晰占据主视区
- 核心信息默认可见
- 侧边栏仅承载次级内容
- MVP 简化未破坏产品核心体验

## 常见反模式
- 首页只有一个空主区域，主要系统全部在侧边栏
- 为了省事，把高频信息做成折叠面板或隐藏抽屉
- 页面布局像后台管理系统，但产品本质是游戏或模拟系统
- 设计文档未明确哪些信息必须常驻显示，导致实现阶段随意塞边栏
